Introduction to Design Patterns

Design patterns are reusable solutions to common problems in software design. They are not direct code but descriptions of how to solve recurring issues, acting as blueprints that can be customized to solve specific problems.

Common Vocabulary

Facilitate communication among developers.

Avoid Reinvention

Prevent solving the same problems repeatedly.

Improve Quality

Lead to more robust and maintainable systems.

What is the Mediator Pattern?

The Mediator pattern is a behavioral design pattern that defines an object (the mediator) that encapsulates how a set of objects (colleagues) interact. Instead of direct communication, colleagues interact through the mediator.

  • Promotes Loose Coupling: Objects don't need to know about each other.
  • Centralizes Logic: Communication logic is managed in one place.
  • Easier Maintenance: Changes to interaction logic are isolated.

Mediator Pattern: Core Concepts

The Intent

To define an object that encapsulates how a set of objects interact. The Mediator promotes loose coupling by keeping objects from referring to each other explicitly, and it lets you vary their interaction independently.

Core Motivation

As a system grows, direct communication between components can create a tangled web of relationships (spaghetti code). The Mediator acts as a central hub, untangling these connections. Instead of many objects talking to many others, they all communicate through one central point, simplifying the system's structure.

When to Use It

  • When a set of objects communicate in complex but well-defined ways.
  • When reusing an object is difficult because it's tightly coupled with many other objects.
  • When a behavior distributed among several classes should be customizable without extensive subclassing.

Why Mediator is Needed?

When objects communicate directly, dependencies proliferate, creating a complex web of connections that becomes a nightmare to manage. The Mediator pattern intervenes to simplify these relationships.

Reduces Tight Coupling

Minimizes direct knowledge objects have of each other.

Simplifies Relationships

Transforms many-to-many into one-to-many.

Cleaner Architecture

Promotes a more organized and understandable system.

Problem Without Mediator

Consider a system with N objects, where each object needs to communicate with every other object. The number of direct communication paths grows exponentially.

  • Maintenance Nightmare: Changing one object requires modifying all related objects.
  • Error Propagation: A change can unexpectedly break functionality elsewhere.
  • Complex Debugging: Tracing communication paths is incredibly difficult.

Structure Part 1: The Mediator

1. Mediator Interface

This is the contract for communication. It defines the methods Colleagues will use to interact with the Mediator.

2. Concrete Mediator

The actual implementation that centralizes communication. It knows about all the Colleagues and contains the logic to route messages between them.

Structure Part 2: The Colleagues

3. Colleague

An interface or abstract class defining how individual objects communicate with the Mediator. Each colleague holds a reference to its mediator.

4. Concrete Colleagues

The actual objects in the system. They send their messages to the Mediator, which then decides what to do. They have no direct knowledge of each other.

Code Example 1: The Participants

First, we define the contract for our mediator (IChatroom) and the concrete implementation for our colleagues (User).

Notice how the User knows about the mediator, but not about other users.

// The Mediator Interface public interface IChatroom { void Register(User user); void SendMessage(string message, User from); } // The Concrete Colleague public class User { public string Name { get; } private IChatroom _chatroom; public User(string name) => Name = name; public void SetChatroom(IChatroom chatroom) => _chatroom = chatroom; public void Send(string message) => _chatroom.SendMessage(message, this); public void Receive(string message) { Console.WriteLine($"{Name} received: {message}"); } }

Code Example 2: The Mediator

Next, the Chatroom class implements the mediator interface. Its main job is to keep a list of users and broadcast messages.

The final part shows how to wire everything up and use it.

// The Concrete Mediator public class Chatroom : IChatroom { private Dictionary<string, User> _users = new(); public void Register(User user) { _users[user.Name] = user; user.SetChatroom(this); } public void SendMessage(string message, User from) { foreach (var user in _users.Values) { if (user != from) { user.Receive(message); } } } } // Usage IChatroom chatroom = new Chatroom(); User user1 = new User("Alice"); User user2 = new User("Bob"); chatroom.Register(user1); chatroom.Register(user2); user1.Send("Hello Bob!"); // Bob receives the message

Advantages

  • Loose Coupling: Colleagues are decoupled from each other, leading to more reusable components.
  • Centralized Control: Interaction logic is in one place, making it easier to understand and manage.
  • Improved Reusability: Individual colleagues can be reused independently of others.
  • Simplified Communication: Reduces complex many-to-many relationships to simple one-to-many.

Disadvantages

  • Mediator Complexity (God Object): The mediator can become overly complex and monolithic if it handles too much logic.
  • Performance Bottleneck: All communication passes through the mediator, which can become a bottleneck in high-throughput systems.
  • Difficult to Maintain: A large, complex mediator can become difficult to maintain over time.

A Concrete Example: GUI Form

Imagine a login dialog. The dialog itself acts as the Mediator. The text boxes, checkbox, and buttons are the Colleagues.

  • The 'Login' button doesn't know about the text boxes. It just tells the dialog "I was clicked".
  • The 'Guest Login' checkbox tells the dialog "I was checked".
  • The dialog (Mediator) then decides what to do. If the checkbox is checked, it might disable the username/password fields. If the login button is clicked, it reads the data from the fields and validates it.
  • This way, the button never talks directly to the textbox. All logic is centralized in the dialog.

Practical Applications

The Mediator pattern is ideal for scenarios with complex communication between multiple objects.

GUI Components

Coordinating interactions between various UI elements like buttons, text fields, and lists in a dialog box.

Chat Applications

A central server (mediator) manages communication between all users (colleagues).

Scheduling Systems

A central scheduler coordinates tasks and resources without them needing to know about each other.

Conclusion

The Mediator pattern is a powerful tool for simplifying complex communication webs, promoting loose coupling and creating more maintainable and scalable systems.
